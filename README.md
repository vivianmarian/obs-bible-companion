# OBS Bible Companion

A system that lets a church operator navigate the Bible and display verses in OBS Studio entirely through [Bitfocus Companion](https://bitfocus.io/companion) button presses — no typing required during a live service.

---

## System Overview

```
Bitfocus Companion (button presses)
        ↕  WebSocket  (port 8765, ?client=companion)
  RelayServer  ←  Node.js sidecar process
        ↕  WebSocket  (port 8765, ?client=browser)
  companion_bridge.js  ←  runs inside OBS Custom Browser Dock
        ↕  BroadcastChannel API (OBS CEF, same instance)
  browser_source.html  ←  OBS Browser Source on scene (verse display)
```

---

## Requirements

- Node.js v20 or later (v24 confirmed working)
- OBS Studio (any recent version)
- Bitfocus Companion v3 (needed from Sprint 6)

## OBS Setup

### Step 1 — Load the plugin UI as a Custom Browser Dock

In OBS: **View → Docks → Custom Browser Docks**

| Field | Value |
|---|---|
| Dock Name | OBS Bible Companion |
| URL | Full path to `project-a/src/index.html` |

> **Important:** This MUST be a Custom Browser Dock, not a Browser Source. Docks stay alive across scene changes. A Browser Source would disconnect the bridge when you switch scenes.

### Step 2 — Add the verse display to your scene

In OBS, on the scene where you want verses to appear:

1. Add a **Browser Source**
2. Set the URL to the full path of `project-a/src/browser_source.html`
3. Set width/height to match your canvas (e.g. 1920×1080)

### Step 3 — Start the RelayServer

```powershell
npm run start:relay --workspace=project-b
```

You should see:
```
[RelayServer] Listening on 127.0.0.1:8765
```

### Step 4 — Reload the OBS Custom Browser Dock

Right-click the dock → **Reload**. The relay terminal should show:
```
[RelayServer] Browser client connected.
```

## Manual End-to-End Test (Node.js scripts)

> **Note:** Earlier drafts of this README recommended `wscat` for manual testing.
> During Sprint 5 testing, `wscat` was found to be unreliable on Windows —
> messages typed interactively were not reliably delivered as WebSocket frames,
> producing no server-side response even under clean, isolated conditions.
> The small Node.js scripts below use the same `ws` library the project
> already depends on, and are confirmed reliable. Use these instead of wscat.

### Prerequisites

- OBS is open with `index.html` loaded as a Custom Browser Dock
- `browser_source.html` is added as a Browser Source on the active scene
- RelayServer is running

### Test 1 — Send a displayVerse command

```powershell
cd project-b/test-scripts
node test-companion.js
```

This connects as `?client=companion`, sends `{"type":"displayVerse","reference":"John 3:16"}`,
and waits for the state response.

**Expected output:**
```
[test-companion] Connected as companion.
[test-companion] SENDING: {"type":"displayVerse","reference":"John 3:16"}
[test-companion] RECEIVED: {"connected":true,"currentReference":"John 3:16","currentTranslation":"KJV","overlayVisible":false,"currentVerseIndex":46}
[test-companion] Test complete — closing.
```

**Expected in OBS:** The dock's search results area shows `John 3:16` as the active verse.
(The overlay is not yet visible on the video — see Test 2.)

### Test 2 — Show the overlay on the video

```powershell
node test-show-overlay.js
```

**Expected output:**
```
[test-show-overlay] SENDING: {"type":"showOverlay"}
[test-show-overlay] RECEIVED: {"connected":true,"currentReference":"John 3:16","currentTranslation":"KJV","overlayVisible":true,"currentVerseIndex":46}
```

**Expected in OBS:** `John 3:16` now appears as an overlay on the video preview/output.

### Test 3 — Simulate the full system without OBS (optional, for relay-only debugging)

Run `test-browser.js` in one terminal (simulates the OBS bridge) and `test-companion.js`
in another. This proves the relay and protocol work correctly in isolation, without
needing OBS open at all — useful for diagnosing whether a problem is in the relay
or specifically in OBS/the bridge.

```powershell
# Terminal A
node test-browser.js

# Terminal B (after Terminal A shows "Connected as browser.")
node test-companion.js
```

## Known Issue and Fix — Blob vs String WebSocket Messages (Sprint 5)

**Symptom:** Commands sent from Companion (or a test script) were received by the
RelayServer and successfully forwarded to the OBS bridge (confirmed via server-side
logging — the browser socket was open and `.send()` was called), but
`companion_bridge.js` never appeared to process them. No errors were logged anywhere.

**Root cause:** OBS's embedded Chromium (CEF) WebSocket implementation can deliver
incoming text frames as a `Blob` object instead of a plain JavaScript string —
different from Node.js's `ws` library (used in automated tests and the manual test
scripts above), which always delivers strings. The original `companion_bridge.js`
contained a defensive guard:

```javascript
if (typeof event.data !== 'string') return
```

This silently discarded every Blob-delivered message with no error, no warning,
and no trace — making it extremely difficult to diagnose. Fake-script-to-fake-script
tests always worked (both ends used `ws`, always strings), while real-OBS tests
always failed identically, with no observable difference in server-side behavior.

**Fix:** `companion_bridge.js` now detects `Blob` data and converts it to text via
`event.data.text()` before parsing, instead of discarding it:

```javascript
ws.onmessage = function (event) {
  if (event.data instanceof Blob) {
    event.data.text().then(function (text) {
      handleIncomingMessage(text)
    })
    return
  }
  handleIncomingMessage(event.data)
}
```

**Lesson for future debugging:** When a message-passing system works perfectly
between two Node.js test scripts but fails silently only when a real browser/CEF
environment is involved, suspect data-type mismatches in WebSocket message
handling first — `Blob`, `ArrayBuffer`, and `string` are all valid `event.data`
types depending on the sender and the receiving environment's `binaryType` setting.

## Security Notes

The RelayServer listens on `127.0.0.1` (localhost) only — it is not reachable from
other machines on the network. No authentication token is implemented. This is
intentional and appropriate for a single church production machine.

## Adding a New Bible Translation

1. Create `project-a/src/bible_data/TRANSLATIONNAME.json`:
   ```json
   [
     { "name": "John 3:16", "verse": "...", "ari": "43:3:16" }
   ]
   ```
2. Regenerate the structure file:
   ```powershell
   npm run generate:metadata --workspace=project-b
   ```
3. Rebuild index.html:
   ```powershell
   npm run build --workspace=project-a
   ```
4. Reload the OBS Custom Browser Dock.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| Dock status shows "Companion: disconnected" permanently | RelayServer not running | Run `npm run start:relay --workspace=project-b` |
| Verse text appears in dock but not on scene | `browser_source.html` not loaded as Browser Source | Add it as a Browser Source on the scene |
| `[RelayServer] ERROR: Port 8765 is already in use` | Another process on port 8765, or a zombie Node process from earlier testing | Run `Get-Process node \| Stop-Process -Force` then restart the relay |
| Commands sent but nothing happens in OBS | Old `companion_bridge.js` without the Blob fix | Ensure you have the Sprint 5 version of `companion_bridge.js` |
| wscat shows `Connected` but typed commands produce no response | wscat reliability issue on Windows | Use `test-companion.js` instead — see Manual End-to-End Test section |
| Translation not persisting after OBS restart | localStorage blocked | Confirm OBS is loading from `file://` path, not `http://` |
